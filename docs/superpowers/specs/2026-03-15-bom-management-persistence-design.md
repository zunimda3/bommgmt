# BOM Management Persistence Design

## Overview

This document defines the approved follow-up design for turning the existing BOM management MVP shell into a real database-backed application while keeping the current demo login flow. The goal of this pass is to replace hardcoded demo data and in-memory state with persistent PostgreSQL storage through Prisma, and to make the existing forms and actions fully operational.

This is not a redesign of the product. The public pages, page structure, navigation, role model, and demo login approach remain in place. The change is in runtime behavior: the app should now save and load real data consistently across sessions and users.

## Goals

- Replace demo and in-memory runtime data with persistent database-backed data
- Make the current forms and actions fully functional for create, edit, and delete workflows
- Enforce role and assignment permissions using real saved records
- Preserve BOM as the source of truth for purchasing
- Keep the demo login flow unchanged for this pass

## Non-Goals

- Replacing demo login with real authentication
- Adding multi-company support
- Redesigning the UI or route structure
- Adding advanced audit history
- Adding external procurement or ERP integrations

## Scope

### Data Types Covered

This persistence pass includes full database-backed behavior for:

- users
- announcements
- projects
- project modules
- BOM items
- purchasing workflow records

### Demo Login

The app continues to use the current demo login model. Users select a demo account on the login page, and the app creates a cookie-backed session. The difference in this pass is that the selected user must exist in the database, and all subsequent access decisions and page data will be based on persisted records.

## Product Behavior

### Users

User management remains owner-only.

The owner can:
- create users
- assign roles
- assign project responsibility through project-level designer and purchaser assignment

This pass does not add advanced user lifecycle features such as password resets, audit logs, or invite emails.

### Announcements

Announcement management is owner-only.

The owner can:
- create announcements
- edit announcements
- delete announcements

All authenticated users can read announcements.

### Projects

Project management remains available to owner and admin.

Owner and admin can:
- create projects
- edit project details
- delete projects
- assign designer and purchaser for each project

Designer and purchaser can only see projects assigned to them.

### Project Modules

Modules remain the grouping structure inside BOM.

Owner, admin, and designer can:
- create modules
- rename modules
- delete modules

Purchaser has read-only access to modules through the BOM view.

### BOM

BOM items remain attached to a specific project module.

Owner, admin, and designer can:
- create BOM items
- edit BOM items
- delete BOM items

Purchaser can view BOM items but cannot change them.

### Purchasing

Purchasing rows remain derived from BOM items and are not manually created as standalone records.

The purchasing list is generated by grouping BOM items within a project by:
- part number
- part description
- vendor
- part category

The generated row stores:
- total quantity
- purchaser workflow fields

Owner, admin, and purchaser can edit only the purchaser-owned workflow fields:
- status
- supplier selected
- quoted price
- PO number
- notes

Designer can view purchasing rows but cannot edit them.

## Data Flow

### Read Flow

All authenticated pages load data from the database:

- dashboard loads announcements and project summaries from persisted records
- announcements page loads saved announcements
- projects page loads visible projects based on role and assignments
- project detail loads project overview, team assignments, modules, BOM items, and derived purchasing rows
- users page loads persisted users

### Write Flow

All form submissions go through server actions backed by Prisma.

Each write action must:
1. resolve the current session user
2. load the relevant database record
3. verify role and project-scope permission
4. validate form input
5. save changes in the database
6. trigger any dependent updates, especially purchasing regeneration after BOM changes

## Purchasing Synchronization Rules

### Source Of Truth

BOM is the source of truth for what must be purchased. Purchasing rows are derived from BOM rather than manually entered.

### Regeneration Behavior

Whenever a BOM item is created, edited, or deleted, the purchasing list for that project is recalculated.

The recalculation must:
- rebuild grouped purchasing totals from current BOM data
- preserve purchaser workflow fields for any row whose aggregate key still matches
- remove purchasing rows that no longer correspond to any grouped BOM data
- create new purchasing rows for newly introduced grouped BOM data

### Aggregate Key

The aggregate key continues to be based on the grouping fields:
- part number
- part description
- vendor
- part category

If those fields change in a way that produces a new aggregate key, prior workflow fields are not reassigned to the new row automatically.

## Permission Model

### Owner

Owner has full access:
- all reads
- all writes
- user management
- announcement management

### Admin

Admin can:
- manage projects
- manage project assignments
- manage modules
- manage BOM
- manage purchasing workflow

Admin cannot manage users or announcements.

### Designer

Designer is restricted to assigned projects and can:
- read assigned projects
- manage modules in assigned projects
- manage BOM in assigned projects
- read purchasing in assigned projects

Designer cannot edit purchasing workflow.

### Purchaser

Purchaser is restricted to assigned projects and can:
- read assigned projects
- read modules and BOM in assigned projects
- edit purchasing workflow in assigned projects

Purchaser cannot edit modules or BOM.

## Validation Rules

The database-backed pass adds server-side validation for all write operations.

Validation includes:
- required text fields
- valid numeric fields for quantity and price
- valid enum values for roles, project status, part category, and purchasing status
- unique project code enforcement
- unique module name within a project
- assignment targets must reference real active users
- purchasing workflow edits must target an existing generated purchasing row

## Error Handling

The system should reject invalid or unauthorized operations in predictable ways:

- unauthenticated users are redirected or blocked from protected pages
- unauthorized users cannot mutate records outside their role or assignment scope
- missing projects, modules, or items return not-found behavior where appropriate
- duplicate project codes and duplicate module names within a project are rejected
- invalid field values are rejected before database writes

## Testing Requirements

This pass should prove the real persisted behavior with automated tests covering:

- database-backed project visibility by role
- owner-only user management
- owner-only announcement CRUD
- owner and admin project CRUD
- owner, admin, and designer module CRUD
- owner, admin, and designer BOM CRUD
- automatic purchasing regeneration after BOM changes
- owner, admin, and purchaser purchasing workflow updates
- role restrictions on all protected actions

## Implementation Direction

The recommended implementation direction is to keep the current routes and screens, then replace the demo data layer with a focused Prisma-backed service layer. Existing server actions should call these database-backed services rather than mutate arrays or in-memory maps. Validation should be added on the server boundary so that page components remain thin and predictable.

This approach upgrades the current application into a real stored system with minimal product disruption and without expanding scope into real authentication.
